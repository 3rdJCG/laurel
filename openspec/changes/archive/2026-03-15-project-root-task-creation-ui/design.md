## Context

`define-task-operations-ui` でサブタスク追加UIが定義されたが、Project 画面でルートタスク（`parentId` なし）を新規作成する操作は仕様化されていない。`subtask-creation` スペックのサブタスク追加フォームと対称的な UI パターンで実装できる見込みがあり、既存の IPC（`data:save-project`）・状態管理パターンもそのまま流用できる。

## Goals / Non-Goals

**Goals:**
- Project 画面からルートタスク（`parentId` なし）を追加できる
- サブタスク追加フォームと一貫した UX（インラインフォーム）を提供する

**Non-Goals:**
- Home 画面でのタスク作成フローへの変更
- genre・tags の初期値設定（作成時はデフォルト値のみ）
- ドラッグ＆ドロップによるタスク並び替え

## Decisions

### 1. 配置：タスク一覧の末尾 vs 先頭 vs フローティングボタン

**決定**: タスク一覧の末尾に「＋ タスクを追加」ボタンを配置し、押下でインライン入力フォームを展開する。

タスク一覧が空の場合は空状態メッセージと共にボタンを目立たせる。フローティングボタン（FAB）は画面レイアウトへの干渉が大きいため採用しない。サブタスク追加と同じインラインフォームパターンを採用することでコードの一貫性を保つ。

### 2. 入力フィールド

**決定**: タイトルのみ入力する最小フォームとする。

ルートタスク作成時に genre・tags を必須にすると入力コストが上がる。作成後に `task-edit` で編集できるため、作成時はタイトルのみに絞る。`Enter` キーで確定、`Escape` キーでキャンセル。

### 3. 状態管理

**決定**: `ProjectScreen` の `useState` でフォーム表示状態とタイトル入力値を管理する。既存の `data:save-project` IPC で永続化する。

## Risks / Trade-offs

- **サブタスク追加フォームとの重複実装** → 将来的にコンポーネント共通化を検討できるが、今回は重複を許容して単純な実装を優先する
