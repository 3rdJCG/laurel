## Context

Home 画面の `define-screen-layout` spec でプロジェクトカードはクリックでプロジェクト画面へ遷移する動作のみが定義されている。プロジェクト名変更・削除のUIを追加するにあたり、カード上でのインタラクションを整理する必要がある。

IPC レイヤーは `implement-data-persistence` で `data:save-project`（更新）・`data:delete-project`（削除）が定義済みで、これをそのまま利用できる。

## Goals / Non-Goals

**Goals:**
- プロジェクトカードからプロジェクト名をインライン編集できる
- プロジェクトカードからプロジェクトを削除できる（確認フロー付き）
- プロジェクト削除時に紐付くタスクもすべて削除される

**Non-Goals:**
- プロジェクトのアーカイブ・非表示
- プロジェクト間のタスク移動
- 削除済みプロジェクトの復元（Undo）

## Decisions

### 1. 編集・削除の操作起点

**決定**: プロジェクトカードにホバー時（またはフォーカス時）に表示される操作メニュー（鉛筆アイコン・ゴミ箱アイコン）を採用する。

カード全体クリックがプロジェクト画面への遷移に割り当てられているため、編集・削除はアイコンボタンから起動する。これにより遷移とCRUD操作の操作起点が明確に分離される。

### 2. プロジェクト名のインライン編集

**決定**: 鉛筆アイコン押下でカード内のプロジェクト名テキストを入力フィールドに切り替えるインライン編集を採用する。

モーダルは使用しない。理由：`define-task-operations-ui` と同様に、画面遷移を増やさず軽量なUXを維持するため。

### 3. 削除確認フロー

**決定**: `define-task-operations-ui` と同様のインライン確認UIをカード内に展開する。

`window.confirm` は使用しない。プロジェクト削除は大量のタスク消失につながるため、確認UIにはプロジェクトに含まれるタスク件数を表示し、影響範囲をユーザーに明示する。

### 4. 連鎖削除の実行場所

**決定**: レンダラー側（React state 更新）で対象プロジェクトと紐付くすべてのタスクを除去してから `data:delete-project` IPC を呼ぶ。

メインプロセス側でタスクファイルを削除する実装は `implement-data-persistence` の `data:delete-project` が担当するため、レンダラーは state のみ更新し永続化はIPCに委ねる。

## Risks / Trade-offs

- **カードホバー操作の発見性** → 初見ユーザーにはアイコンが隠れていることが分かりにくい可能性がある。常時表示も検討できるが、現時点ではシンプルさを優先してホバー表示を採用
- **タスク件数表示のコスト** → 削除確認時にタスク件数を表示するには state からカウントが必要だが、既にメモリ上にロード済みのデータを使えるため追加I/Oは不要
