## Context

Laurel は Electron アプリ。レンダラー（React）はファイルシステムに直接アクセスできず、contextBridge 経由の IPC でメインプロセスと通信する。`implement-data-persistence` で `data:load-all` IPC チャンネルが定義されたが、呼び出しタイミングと責務が未定義のまま。また、Home 画面のプロジェクト作成 UI の形式と、Home/Project 両画面でのタスク追加責務が未整理である。

## Goals / Non-Goals

**Goals:**
- `data:load-all` をいつ・どこから呼ぶか明確にする
- ローディング状態とエラー状態をどのコンポーネントが持つか決める
- プロジェクト作成 UI のインタラクション（ボタン位置・フォーム形式・操作フロー）を決める
- Home 画面でのタスク追加要件を削除し、タスク追加の責務を Project 画面に統一する

**Non-Goals:**
- `data:load-all` の実装詳細（メインプロセス側の処理は `implement-data-persistence` のスコープ）
- プロジェクト名のバリデーション詳細（空文字チェックのみ）
- プロジェクト作成後のナビゲーション動作（Home 画面に留まる）

## Decisions

### 1. データロードはレンダラーが起動時に invoke する

**決定:** アプリ起動時、レンダラーの最上位コンポーネント（`App.tsx` またはデータ管理用 Provider）の `useEffect([], [])` で `window.api.invoke('data:load-all')` を呼ぶ。

**理由:**
- レンダラーが自分のデータを必要なタイミングで取りに行く Pull 型は、React の宣言的なデータフローと相性が良い
- メインプロセスが起動後に `send` する Push 型では、レンダラーのマウント完了前にイベントが届いてしまうレースコンディションが発生しうる
- `useEffect([], [])` はマウント後に1回だけ実行されるため、二重呼び出しが発生しない（React 18 の StrictMode では開発時に2回実行されるが、冪等な read 操作なので問題なし）

**代替案（却下）:** メインプロセスが `did-finish-load` 後に `send` → レンダラーが `on` で受け取る。レースコンディションを防ぐためにキューイングが必要になり複雑化する。

### 2. ローディング状態はデータ管理コンテキストが管理する

**決定:** `App.tsx` に近い位置にデータ管理用の Context/Provider を配置し、`isLoading: boolean` と `error: string | null` を状態として持つ。ロード中は全画面ローディング表示、エラー時はエラーメッセージを全画面表示する。

```
App
└── DataProvider  ← useEffect で data:load-all を invoke、isLoading/error を管理
    ├── (isLoading) → <LoadingScreen />
    ├── (error)     → <ErrorScreen message={error} />
    └── (ready)     → <MainLayout />  ← 通常の画面レイアウト
```

**理由:** 各画面コンポーネントが個別にローディング状態を管理すると重複が生まれる。データ取得は1度だけ行うため、最上位で集中管理するのが適切。

### 3. プロジェクト作成 UI はインラインフォーム展開方式

**決定:** Home 画面のプロジェクト一覧の末尾に「＋ プロジェクトを追加」ボタンを配置し、押下するとインラインのプロジェクト名入力フォームが展開される。`project-root-task-creation-ui` と同じインタラクションパターンを採用する。

**インタラクション:**
- 「＋ プロジェクトを追加」ボタン → フォーム展開
- Enter キーまたは確定ボタン → プロジェクト作成（空文字の場合は作成しない）
- Escape キーまたはキャンセルボタン → フォームを閉じる（作成しない）
- プロジェクト作成後 → フォームを閉じ、作成したプロジェクトが一覧末尾に表示される

**ボタン配置:** Home 画面本体（サイドバーではなく、コンテンツエリアのプロジェクト一覧の下）

**理由:**
- `project-root-task-creation-ui` と同じパターンにすることでコンポーネント構造を再利用しやすくなる
- インラインフォームはモーダルより操作ステップが少ない（ダイアログが開かない）
- サイドバーでのプロジェクト作成は `navigation-sidebar` のスコープ外のため、Home 画面本体に配置

**代替案（却下）:** モーダルダイアログ → 操作ステップが増え、インラインフォームで十分な単純入力にはオーバーキル。

### 4. Home 画面でのタスク追加要件を削除

**決定:** `home-screen` 仕様から「タスクの新規追加とプロジェクトへの分類」要件（Scenario 含む）を削除する。タスク追加は Project 画面（`project-root-task-creation-ui`）のみに統一する。

**理由:**
- Home 画面でのタスク追加にはプロジェクト選択 UI が必要で、実装コストが高い
- ユーザーは Project 画面に遷移してからタスクを追加するフローで一貫性が生まれる
- MVP 段階では機能を絞り、Home 画面はプロジェクト一覧・選択・作成に集中する
- 将来的に「インボックス的なタスク追加」が必要になった場合は別 Change で追加できる

## Risks / Trade-offs

- [Home からタスクを素早く追加したいユーザーには不便] → MVP 段階では Project 画面経由を徹底。将来 Change で対応を検討
- [React 18 StrictMode での useEffect 二重実行] → `data:load-all` は冪等な読み取りのため問題なし。書き込み系 IPC は呼ばない
- [data:load-all が失敗した場合] → `error-handling-ui` の仕様に従いエラー表示。本 Change ではエラー状態の存在を定義するのみ
