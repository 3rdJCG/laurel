## Context

`define-project-task-model` でタスクデータモデルが確定し、`define-screen-layout` でプロジェクト画面（タスク一覧）が定義された。しかしタスクの操作UI（ステータス変更・編集・削除・サブタスク追加）はどちらのchangeにも含まれておらず、実装の指針がない状態にある。

データ永続化は `implement-data-persistence` が担当し、保存には既存の `data:save-project` IPC チャンネルを使用できる。

## Goals / Non-Goals

**Goals:**
- プロジェクト画面のタスクアイテムからステータスをインラインで変更できる
- タスクのタイトル・genre・tagsをインライン編集できる
- タスクを削除できる（誤削除を防ぐ確認ステップ付き）
- 既存タスクの直下にサブタスクを追加できる（5層制限のUIガード付き）

**Non-Goals:**
- タスクの並び替え（ドラッグ＆ドロップ）
- タスクのフィルタリング・検索
- タスク詳細専用画面の新設
- プロジェクト画面以外での操作（Home画面でのタスク編集等）

## Decisions

### 1. インライン編集 vs モーダル編集

**決定**: インライン編集を採用する。

タスク一覧アイテム上で直接編集状態に切り替える方式（ダブルクリックまたは編集ボタン押下）。モーダルは画面遷移が多くなりUXが重くなるため採用しない。

### 2. ステータス変更のUI方式

**決定**: ステータスバッジのクリックによるセレクト/ドロップダウン方式を採用する。

`todo` / `in-progress` / `in-review` / `done` の4値を視覚的に識別できるバッジで表示し、クリックで選択肢を切り替える。ステータス順次サイクル（クリックで次へ）ではなく、任意の値に一発で変更できるドロップダウンとする。理由：`done` から `todo` に戻すユースケースがあるため。

### 3. 削除確認フロー

**決定**: インライン確認（"本当に削除しますか？ [削除] [キャンセル]"）をタスクアイテム内に展開する方式とする。

OS ダイアログ（`window.confirm`）は使用しない。理由：Electron 環境でのスタイル統一と、誤タップ防止のためにフォーカスを奪わない方式を優先する。

### 4. サブタスク表示とネスト

**決定**: プロジェクト画面でサブタスクをインデントして親タスクの直下に表示する。

サブタスク追加ボタンは各タスクアイテムに設ける。5層目のタスクにはサブタスク追加ボタンを表示しないことで、6層目作成を防止する（UIレベルのガード）。

### 5. 状態管理

タスク操作後の状態更新は React の `useState` で管理するローカルステートを直接更新し、`data:save-project` IPC で永続化する。グローバルストア（Redux等）は導入しない（過剰設計）。

## Risks / Trade-offs

- **インライン編集の複雑さ** → 編集状態を `editingTaskId` のような単一 state で管理し、同時に複数タスクを編集中にならないよう制御する
- **サブタスクの再帰レンダリング** → 最大5層のため再帰コンポーネントを使うが、深さを `depth` props で伝播させることで制御可能
- **削除時の子タスクの扱い** → 親タスク削除時は子タスクも連鎖削除する（孤立タスク防止）。specに明記する
