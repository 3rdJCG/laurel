## Context

`implement-data-persistence` の IPC 設計では `data:load-all`・`data:save-project` の成功パスのみを定義している。失敗パス（ファイルアクセス拒否・ネットワーク切断・JSON 破損等）についてはリスクセクションに「エラーをレンダラーに通知」と書かれているだけで、通知UIの形式・挙動・復帰フローが未定義。

## Goals / Non-Goals

**Goals:**
- 書き込み失敗・読み込み失敗・JSON 破損の3ケースのUI挙動を定義する
- エラーはユーザーが認知できる形で表示し、操作継続可能にする
- エラー状態からの復帰（再試行・無視・フォールバック）フローを定義する

**Non-Goals:**
- エラーログの永続化・レポート機能
- 競合解決UI（将来の change で対応予定）
- ネットワーク復帰の自動検知

## Decisions

### エラー通知UI：非モーダルのエラーバナー（dismissible）

画面上部または下部に固定表示するバナー形式を採用する。モーダルダイアログは操作をブロックするため、重大度が低〜中のエラーには適さない。

**エラーの重大度分類**:
- **Critical（保存失敗）**: 赤バナー + 「再試行」ボタン。データロストの可能性があるため目立つ表示。
- **Warning（読み込み時の一部ファイルスキップ）**: 黄バナー + 「詳細を見る」リンク。アプリ操作は継続可能。

**代替案**: トースト通知（自動消去）は保存失敗のような重要なエラーで見落とされるリスクがあるためリジェクト。常に目に入るバナーを採用。

### 破損JSON：スキップ＋警告表示

`data:load-all` 時に parse 失敗したファイルはスキップし、他のプロジェクトは正常に読み込む。スキップしたファイルのリストをレンダラーに返し、Warning バナーで通知する。

**復帰オプション**: ファイルパスを表示し「ファイルを開く」ボタン（`shell.showItemInFolder`）で手動復旧を可能にする。

**代替案**: 破損ファイルを検出したらアプリ起動をブロックする案は、1つのファイル破損で全データにアクセスできなくなるためリジェクト。

### IPC エラーレスポンス形式

既存 IPC のレスポンスに `{ error?: { code: string; message: string; filePath?: string } }` を追加する。成功時は `error` フィールドなし。

## Risks / Trade-offs

- **バナーの重複表示**: 複数エラーが連続する（ネットワーク断等）とバナーが積み重なる可能性。→ 同種エラーは集約して1件のバナーに表示する。
- **再試行ループ**: 保存失敗後に再試行ボタンを押しても同じエラーが続く場合がある。→ 再試行は1回のみ自動で行い、それ以降はユーザー操作を待つ。
