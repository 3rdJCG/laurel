## Context

現在の `TaskDetailScreen` は「ルートタスク＋子タスクのリスト」を `TaskItem` コンポーネントで再帰的に表示している。`TaskItem` は深さ5まで再帰ネストを許容し、ドラッグ&ドロップ、インライン編集、タグ候補、日付入力など多数のUI要素を一つのコンポーネントに集約している。また `TaskDetailTab` という別のツリー表示コンポーネントも存在し、重複した責務を持つ。

結果として：
- タスク1件あたりのUI要素が多く（ドラッグハンドル、展開ボタン、ジャンル、ステータス、期限、タグ、アクションボタン群）、視覚的に重い
- 階層の深さに制限がなく、ユーザーの意図しない深いネストが発生しうる
- `TaskDetailTab` と `TaskDetailScreen` の役割が重複している

## Goals / Non-Goals

**Goals:**
- タスク詳細画面をシンプルな「タスク情報ヘッダー ＋ 子タスクKanban」構成に再設計する
- 子タスクの階層を1層（ルートタスクの直接の子のみ）に制限する
- Kanbanビューでステータス別列（TODO / 進行中 / レビュー中 / 完了）に子タスクを表示する
- `TaskDetailTab` を廃止し画面構造を整理する

**Non-Goals:**
- プロジェクト一覧画面（`ProjectScreen`）のルートタスクリスト表示は変更しない
- Issues タブの変更は行わない
- Kanban内のカード間ドラッグ&ドロップ（列をまたぐ移動）は今回スコープ外（ステータス変更はクリック操作のみ）

## Decisions

### 1. TaskDetailScreen の構成を「タスクヘッダー ＋ Kanban」に再設計する

**決定**: タブ構造（Tasks / Issues）は維持しつつ、Tasks タブの内容をKanbanボードに置き換える。ルートタスクの詳細情報（タイトル・ジャンル・タグ・日付）はタブ上部にコンパクトなヘッダーとして表示する。

**根拠**: タブ構造を維持することで Issues タブとの共存が自然。タスク情報ヘッダーをKanbanの上に置くことで「このタスクに属する子タスクを管理する」という文脈が明確になる。

**代替案**: Tasks/Kanban を切り替えトグルで提供する案は、切り替え先が存在しないため不要。

### 2. 新コンポーネント KanbanView を作成する

**決定**: `src/renderer/src/components/KanbanView.tsx` を新規作成する。4列（todo / in-progress / in-review / done）の固定列構成とし、各列に子タスクのカードを表示する。

**根拠**: `TaskItem` は多機能すぎてKanbanカードとして再利用するとオーバーエンジニアリングになる。軽量な `KanbanCard` を `KanbanView` 内部に定義し、必要最小限の情報（タイトル・ステータスバッジ・削除ボタン）のみ表示する。

**カード上のUI要素**:
- タスクタイトル（クリックで即インライン編集）
- ステータスは列で表現するため不要（列間移動ボタンまたは省略）
- 削除ボタン

### 3. 1階層制限はUIのみで強制する

**決定**: データモデル（Task型）は変更せず、UIレベルでサブタスク追加ボタンを子タスクカードに表示しないことで1階層を強制する。既存データへのマイグレーションは行わない。

**根拠**: データモデルを変更すると既存データへの影響が大きい。既存の深いネストデータは `TaskDetailScreen` に遷移した際には孫タスクが表示されなくなるが、データ自体は保持される。`ProjectScreen`（ルートタスクリスト）では引き続き `TaskItem` がツリー表示するため、既存の深いネストもそちらから確認可能。

**代替案**: `createTask` に parentId のバリデーションを追加してDBレベルで防止する案もあるが、既存データとの整合性問題が生じるため見送り。

### 4. TaskDetailTab を廃止し、タスク情報をヘッダーに統合する

**決定**: `TaskDetailTab.tsx` を削除し、タスク情報（タイトル・ジャンル・タグ）の表示・編集は `TaskDetailScreen` 内の `TaskInfoHeader` コンポーネント（同ファイル内またはインライン）に統合する。

**根拠**: `TaskDetailTab` と `TaskDetailScreen` の `TaskItem` 表示は責務が重複しており、どちらかに集約すべき。ヘッダーに統合することで画面の流れが「タスク情報 → 子タスク管理」と明確になる。

## Risks / Trade-offs

- **既存の深いネストデータが非表示になる**: Kanban画面では孫タスク以降が見えなくなる。ただし `ProjectScreen` のリスト表示からは引き続きアクセス可能。[Risk] → ユーザーへのガイドとして、将来的にデータ移行ツールを検討できるが今回は対象外
- **KanbanCard が TaskItem より機能が少ない**: 日付・タグ表示がカードから省かれる場合、情報が見づらくなる可能性。[Risk] → カードにタグ・期限日のバッジ表示を追加することで最小限の情報を確保する
- **Kanban内のドラッグ&ドロップなし**: 順序変更ができないことでユーザーが不満を感じる可能性。[Risk] → 今回は列内での並び替えドラッグ&ドロップを実装してステータス列の絞り込み管理に集中させる（列をまたぐドラッグは次フェーズ）

## Migration Plan

1. `KanbanView` コンポーネントを新規作成する
2. `TaskDetailScreen` を新構成（タスクヘッダー ＋ KanbanView）に書き換える
3. `TaskDetailTab.tsx` を削除する（参照箇所がなくなってから）
4. `TaskItem` のサブタスク追加ボタンのdepth制限を `depth === 0`（ルートのみ）に変更する

既存データへのマイグレーションは不要。ロールバックはgit revertで対応可能。

## Open Questions

- Kanbanカードのタイトルクリック時の遷移先: 子タスクは詳細画面を持たない設計にするか、それとも `TaskDetailScreen` を再帰的に開くか → 子タスクの詳細は今回はインライン編集のみとし、サブ画面遷移はなし
- 子タスクのない列は表示するか折り畳むか → 常に4列表示（空列も表示）とし、一貫性を保つ
